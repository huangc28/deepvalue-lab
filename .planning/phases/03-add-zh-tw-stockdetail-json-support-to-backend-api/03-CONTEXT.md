# Phase 3: Add zh-TW StockDetail JSON support to backend API - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning
**Source:** PRD Express Path (plan file — PRD: Locale-aware Stock Detail API)

<domain>
## Phase Boundary

This phase adds zh-TW locale support to the backend stock detail API layer. The skill (Phase 2) already produces two monolingual StockDetail JSONs (EN and zh-TW). This phase makes the backend accept, store, and serve the zh-TW JSON so the frontend can request locale-appropriate data.

Scope: backend only (Go service, Turso DB, R2 storage). No frontend changes. No generic i18n framework — zh-TW only.

</domain>

<decisions>
## Implementation Decisions

### Data Model
- Add `r2_detail_zh_tw_key TEXT NOT NULL DEFAULT ''` to `published_stock_details`
- Add `summary_json_zh_tw TEXT NOT NULL DEFAULT '{}' CHECK (json_valid(summary_json_zh_tw))` to `published_stock_details`
- Safe defaults mean no backfill needed for existing rows

### R2 Key Convention
- EN detail key: `reports/{ticker}/{YYYYMMDD}/{reportID}.json` (unchanged)
- zh-TW detail key: `reports/{ticker}/{YYYYMMDD}/{reportID}.zh-TW.json`

### Publish API (POST /v1/stocks/{ticker}/reports)
- Add optional `stockDetailZhTW` field (json.RawMessage, nil = not provided)
- When present: upload to R2, extract summary, persist both zh-TW columns
- When absent: backward compatible (zh-TW columns stay at defaults)
- Response includes `r2DetailZhTWKey` when zh-TW was provided

### Detail API (GET /v1/stocks/{ticker})
- Read `?locale=zh-TW` query param
- If zh-TW requested and `r2_detail_zh_tw_key != ''`: serve zh-TW from R2
- If zh-TW requested but no R2 key: fall back to `summary_json_zh_tw`, then EN
- Default (no locale param): EN behavior unchanged

### List API (GET /v1/stocks)
- Read `?locale=zh-TW` query param
- If zh-TW and `summary_json_zh_tw` is non-empty/non-`{}`: use it
- Otherwise: fall back to EN summary

### Backward Compatibility
- `stockDetailZhTW` is optional — old callers unaffected
- `?locale` defaults to EN — existing GET behavior unchanged
- Pre-existing rows have safe defaults — handlers fall back to EN gracefully

### Claude's Discretion
- Migration file timestamp (use current time)
- sqlc regeneration mechanics
- Error message wording for zh-TW upload failures
- Whether to validate zh-TW StockDetail fields (same minStockDetail check as EN, or lighter)

</decisions>

<specifics>
## Specific Ideas

- The `SELECT *` queries (GetPublishedStockDetail, ListPublishedStockDetails) will automatically pick up new columns after sqlc regeneration — no SQL changes needed for reads
- Only UpsertPublishedStockDetail needs SQL modification (INSERT columns + ON CONFLICT SET)
- zh-TW summary extraction reuses the same `summaryFields` struct unmarshal pattern as EN
- R2 key helper follows existing `DetailKey()` pattern with `.zh-TW.json` suffix
