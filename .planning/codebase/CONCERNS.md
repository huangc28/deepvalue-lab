# Codebase Concerns

**Analysis Date:** 2026-03-19

## Tech Debt

**No explicit error handling for ignored query errors:**
- Issue: `be/lib/app/subscriptions/create_handler.go` line 46 intentionally ignores the error from `GetSubscription()` with `existing, _ := h.queries.GetSubscription(...)`. This assumes the query always succeeds or fails gracefully, but doesn't validate.
- Files: `be/lib/app/subscriptions/create_handler.go` (line 46)
- Impact: If `GetSubscription` fails for unexpected reasons, the handler still proceeds to create/update the subscription but the status code logic relies on a potentially zero-valued struct, which could return `http.StatusOK` incorrectly when it should be `http.StatusCreated`.
- Fix approach: Explicitly handle the error, distinguish between "not found" (expected) and other errors (unexpected), or add logging to understand failure modes.

**Optional nil return in Redis client initialization:**
- Issue: `be/cache/redis.go` returns `nil, nil` when `RedisURL` is empty, which is a pattern that defers configuration validation to runtime. This is a convenience pattern but creates a silent degradation mode.
- Files: `be/cache/redis.go` (line 10)
- Impact: Code that uses the Redis client must check for nil before calling methods. Currently no callers are present, but future integration could silently fail to cache if the env var is missing.
- Fix approach: Either fail fast on missing `REDIS_URL` if Redis is required, or document the optional behavior and ensure all callers null-check the client.

**Optional nil return in SQLite client initialization:**
- Issue: `be/lib/pkg/sqlite/client.go` returns `&Client{Enabled: false}` when DSN is not configured. This creates a pattern where downstream code must check `Enabled` before using the database.
- Files: `be/lib/pkg/sqlite/client.go` (lines 29-30)
- Impact: The `provideTursoQueries()` in `be/lib/app/fx/core.go` line 31 doesn't check if `client.DB` is nil before passing to `turso_models.New()`. If database initialization is skipped but FX tries to inject queries anyway, this will panic.
- Fix approach: Either validate in FX that if queries are required, the client must be enabled, or add defensive nil checks in `provideTursoQueries()`.

**Generated code in hot path:**
- Issue: `be/lib/app/turso_models/stocks.sql.go` is auto-generated by sqlc (230 lines). Any changes to the database schema require re-running code generation, which is an extra manual step that could be forgotten.
- Files: `be/lib/app/turso_models/stocks.sql.go`
- Impact: Schema changes may not be reflected in generated code, causing runtime errors or type mismatches.
- Fix approach: Add pre-commit hook or CI check to ensure sqlc is run after schema changes. Document the generation process in README.

## Known Bugs

**Missing detail artifact returns 404:**
- Symptoms: Stock detail endpoint returns `404` when neither locale-specific nor English detail artifacts are available.
- Files: `be/lib/app/stocks/detail_handler.go`
- Trigger: Any stock record in `published_stock_details` without a usable `r2_detail_key`, or a locale-specific request where the zh-TW key is absent and the English key is also absent.
- Current behavior: This is the intended contract. The detail endpoint is detail-only and does not fall back to `summary_json`.
- Operational implication: Publish flows should ensure `r2_detail_key` is populated for any ticker expected to render in the detail page.

## Security Considerations

**R2 credentials in config:**
- Risk: R2 access keys (AccountID, AccessKeyID, SecretAccessKey) are passed as plain environment variables and stored in the Config struct. They're never explicitly cleared from memory.
- Files: `be/config/config.go` (lines 24-30), `be/lib/pkg/r2/client.go` (lines 37-41)
- Current mitigation: Credentials are only used to initialize the S3 client at startup, not logged or exposed in response payloads.
- Recommendations: Use environment variable redaction in logging (zap does this with sensitive fields). Consider using cloud provider's managed identities (Azure Managed Identity, AWS IAM Roles) instead of explicit keys if running on managed infrastructure. Never log or dump the Config struct.

**Database connection string in config:**
- Risk: Turso token and DSN are stored in plain Config struct. If config is accidentally logged or dumped, credentials leak.
- Files: `be/config/config.go` (lines 17-22), `be/lib/pkg/sqlite/client.go` (lines 23-36)
- Current mitigation: No code currently logs the full config or DSN.
- Recommendations: Add struct tag `masked:"true"` to sensitive fields or implement custom String() method that redacts them. Audit for any code that might stringify Config.

**JSON.RawMessage pass-through in publish handler:**
- Risk: `be/lib/app/stocks/publish_handler.go` lines 36 and 115 accept and upload arbitrary JSON without validation. Malicious client could inject embedded scripts or excessively large payloads.
- Files: `be/lib/app/stocks/publish_handler.go` (lines 36, 115)
- Current mitigation: Request body size limits are typically enforced at reverse proxy/server level, not in code.
- Recommendations: Add content-length limit in handler or middleware. Validate uploaded JSON structure against expected schema before upload to R2.

## Performance Bottlenecks

**SQLite connection pool tuned for single-connection Turso:**
- Problem: `be/lib/pkg/sqlite/client.go` lines 62-64 set `MaxOpenConns=1, MaxIdleConns=1, ConnMaxLifetime=1min`. This is appropriate for Turso's libsql client (which manages its own pooling), but could be a bottleneck if this library is ever repurposed for local SQLite.
- Files: `be/lib/pkg/sqlite/client.go` (lines 62-64)
- Cause: libsql is a remote-backed SQLite driver. Connection pooling at sqlx level is redundant and can cause unnecessary queueing.
- Improvement path: If switching to local SQLite or PostgreSQL, adjust pool settings. Consider adding a tuning parameter in Config.

**Mock data size:**
- Problem: `web/src/data/mock-stocks.ts` is 1765 lines for a single stock's mock data (TSM). This will scale linearly with more stocks.
- Files: `web/src/data/mock-stocks.ts`
- Cause: Each stock has nested arrays of scenarios, news events, and fact items all inlined.
- Improvement path: Break mock data into separate files by stock. Consider lazy-loading or pagination once backend API is integrated. Performance is not critical for frontend mock, but maintainability will suffer.

**Dashboard filters use linear scan:**
- Problem: `web/src/pages/dashboard-page.tsx` lines 47-65 filter `mockStocks` with `.filter().toSorted()` on every render. With hundreds of stocks, this could become noticeable.
- Files: `web/src/pages/dashboard-page.tsx` (lines 47-65)
- Cause: Frontend mocking doesn't use indexed queries or materialized filters.
- Improvement path: Once backend API exists, move filtering and sorting to server. Client-side: consider memoizing filter/sort results more aggressively.

## Fragile Areas

**Turso queries assume DB is ready:**
- Files: `be/lib/app/fx/core.go` (line 31)
- Why fragile: `provideTursoQueries()` creates `turso_models.Queries` without checking if `client.DB` is non-nil. If `Enabled=false`, this will panic when any handler tries to call a query method.
- Safe modification: Add a validation step in FX that ensures either (1) Turso is configured, or (2) handlers that need Turso are not registered. Alternatively, wrap all query calls with a nil-check or panic with a clear error message.
- Test coverage: No test coverage for initialization with missing Turso config.

**Request body decoding without content-type validation:**
- Files: `be/lib/app/stocks/publish_handler.go` (line 80), `be/lib/app/subscriptions/create_handler.go` (line 35)
- Why fragile: Both handlers use `json.NewDecoder(r.Body).Decode()` without verifying `Content-Type: application/json` header. If a malformed request arrives, error message is generic.
- Safe modification: Add middleware to validate content type, or add explicit checks in handler. Document expected request format.
- Test coverage: No tests for malformed content-type scenarios.

**R2 upload failure leaves database in inconsistent state:**
- Files: `be/lib/app/stocks/publish_handler.go` (lines 108-119)
- Why fragile: Sequence is: upload markdown → upload detail JSON → insert report → upsert stock detail. If upload succeeds but database insert fails, R2 has orphaned artifacts with no record in DB. Reverse is also true: if R2 fails, DB transaction still commits.
- Safe modification: Either wrap all operations in a transaction (if Turso supports it), or implement a cleanup/retry mechanism. Consider idempotency: if the same report is published twice, ensure it's safe.
- Test coverage: No tests for partial failure scenarios.

**Config validation happens too late:**
- Files: `be/config/config.go` (lines 58-64), `be/lib/pkg/r2/client.go` (lines 23-33)
- Why fragile: R2 client validates credentials in `NewClient()` at startup. If credentials are wrong, the app crashes at boot with a generic error message. No health check verifies R2 is actually accessible.
- Safe modification: Add explicit config validation in a startup hook. Test R2 connectivity in health check. Fail fast with clear error messages.
- Test coverage: No tests for invalid config scenarios.

**Type safety gap for LocalizedText round-tripping:**
- Files: `be/lib/app/stocks/publish_handler.go` (line 60), `web/src/types/stocks.ts` (various LocalizedText fields)
- Why fragile: Frontend sends `LocalizedText` (typed as `{ en: string, 'zh-TW': string }`) to backend, backend stores it as raw JSON in `summary_json`, then returns it as JSON. If frontend schema changes (adds new language), database doesn't know about it and returns stale keys.
- Safe modification: Version the JSON schema. Validate round-tripped data against expected type structure. Add migration path for old data.
- Test coverage: No tests for schema evolution or missing localization keys.

## Scaling Limits

**Single R2 account for all regions:**
- Current capacity: One Cloudflare R2 bucket with global replication.
- Limit: If traffic to reports reaches tens of millions of requests/month, latency could degrade for non-US regions. R2 replicates globally but has eventual consistency windows.
- Scaling path: Implement CDN caching layer (Cloudflare Cache) or multi-region R2 buckets. Monitor access patterns and implement geographic routing.

**Turso connection limit with max=1:**
- Current capacity: Single active connection to Turso at a time. Handles sequential queries fine, but concurrent requests queue.
- Limit: If sustained load exceeds ~100 req/sec with non-trivial query latency, queueing becomes noticeable.
- Scaling path: If higher load needed, switch to PostgreSQL backend or implement query batching. For now, this is appropriate for a research tool.

**Frontend mock data in-memory:**
- Current capacity: 1765-line mock-stocks.ts bundled in web frontend. Serves ~3-5 stocks depending on mock coverage.
- Limit: ~10-20 stocks before bundle size and initial parse time becomes notable. No pagination.
- Scaling path: Connect to backend API instead of mocks. Implement pagination on backend. Consider lazy-loading detail views.

## Dependencies at Risk

**Turso libsql-client-go version mismatch:**
- Risk: Go module uses `github.com/tursodatabase/libsql-client-go v0.0.0-20251219100830-...` (pre-v1.0 with pinned commit hash). No stable version exists.
- Impact: If Turso releases breaking changes or deprecates the driver, no migration path other than manual update.
- Migration plan: Monitor Turso releases. Consider building on top of libsql's HTTP API directly if client becomes unmaintained. Eventually move to stable v1.0 once released.

**AWS SDK v2 in beta/rapidly evolving:**
- Risk: `github.com/aws/aws-sdk-go-v2` major versions frequently introduce breaking changes.
- Impact: Security updates may require code changes. Current v1.41.4 is recent but not guaranteed stable.
- Migration plan: Pin to stable minor versions. Test major version upgrades before deploying. Consider abstracting S3 client behind an interface for easier swaps.

**Go 1.25.0 (very recent, potential stability issues):**
- Risk: Project uses `go 1.25.0`, which is bleeding-edge. Not all dependencies may be tested against it.
- Impact: Potential edge case bugs or incompatibilities with third-party packages.
- Migration plan: Wait for 1.25.1 or later patch release before production. Use Go 1.24.x LTS for production deployments.

## Missing Critical Features

**No authentication or authorization:**
- Problem: All API endpoints (`/v1/stocks/*`, `/v1/subscriptions`, etc.) are wide open. Any client can publish reports or subscribe to tickers.
- Blocks: Cannot safely deploy to production without auth. Need to control who can publish reports and manage subscriptions.
- Workaround: Reverse proxy with IP whitelist or API key validation. Not implemented in code.

**No rate limiting:**
- Problem: Endpoints have no per-client rate limits. Malicious actor could flood R2 with uploads or spam the database.
- Blocks: Not production-ready without rate limiting middleware.
- Workaround: Implement rate limiting at reverse proxy level (nginx, Cloudflare).

**No graceful handling of R2 connectivity loss:**
- Problem: If R2 becomes unreachable mid-request, the handler fails and leaves database records without corresponding R2 artifacts.
- Blocks: Not suitable for high-availability deployment.
- Workaround: Implement retry logic with exponential backoff. Add dead-letter queue for failed uploads.

**No data validation before storage:**
- Problem: Publish handler accepts arbitrary `StockDetail` JSON without schema validation. Could store malformed or incomplete data.
- Blocks: Data quality degrades over time if clients send incomplete records.
- Workaround: Implement JSON Schema validation or use typed struct unmarshaling with strict checks.

**No pagination on list endpoints:**
- Problem: `/v1/stocks` and `/v1/stocks/{ticker}/reports` return all results. If 1000+ stocks are published, response becomes large and slow.
- Blocks: Doesn't scale to large watchlists.
- Workaround: Client-side filtering. Eventually implement server-side pagination.

**No API versioning strategy:**
- Problem: All endpoints use `/v1/` prefix, but there's no versioning convention or deprecation policy. If schema changes, clients break.
- Blocks: Cannot evolve API without coordinating all clients.
- Workaround: Document that `/v1/` is unstable. Plan for `/v2/` with breaking changes.

## Test Coverage Gaps

**No unit tests for handlers:**
- What's not tested: Request/response logic, error cases, edge cases in publish flow.
- Files: `be/lib/app/stocks/publish_handler.go`, `be/lib/app/stocks/detail_handler.go`, `be/lib/app/subscriptions/create_handler.go`, `be/lib/app/stocks/list_handler.go`, `be/lib/app/stocks/reports_list_handler.go`
- Risk: Silent failures if handler logic changes. Error cases (malformed requests, database failures) are never validated.
- Priority: High - handlers are critical code path.

**No integration tests:**
- What's not tested: End-to-end flows like "publish report → fetch detail → list reports". Database and R2 integration untested.
- Files: All files in `be/lib/app/`
- Risk: Database schema mismatches, R2 key format issues, or transaction failures won't be caught until production.
- Priority: High - these are the main use cases.

**No frontend component tests:**
- What's not tested: React components rendering correctly with various data states. Mock data edge cases.
- Files: `web/src/components/`, `web/src/pages/`
- Risk: UI breaks with edge cases (missing fields, malformed data, null values).
- Priority: Medium - frontend is not yet production-critical but matters for user experience.

**No E2E tests:**
- What's not tested: Complete user workflows from frontend to backend to R2.
- Risk: Frontend → backend integration issues (type mismatches, missing fields) undetected.
- Priority: Medium - relevant once frontend connects to real backend.

**No load/stress tests:**
- What's not tested: Behavior under high request volume. Database connection queueing. R2 rate limits.
- Risk: Performance issues or crashes only discovered in production.
- Priority: Low for MVP, high before scaling.
